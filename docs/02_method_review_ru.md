IBP-редукция в параметрическом представлении: разбор метода и рабочая интерпретация
================================================================================

Версия: 2026-07-06
Назначение файла: зафиксировать метод, его математический смысл и практические правила, которые должны лечь в основу автоматического Python-редуктора.


1. Общая постановка
-------------------

Рассматривается семейство параметрических интегралов вида

    J(n,m) = ∫_{R_+^N} dx_1 ... dx_N
             prod_i x_i^(nu_i + n_i - 1)
             prod_l G_l(x)^(mu_l + m_l),

где

    n = (n_1,...,n_N) ∈ Z^N,
    m = (m_1,...,m_M) ∈ Z^M,
    G_l(x) — полиномы по параметрам x_i,
    nu_i, mu_l могут зависеть от epsilon и внешних параметров,
    коэффициенты G_l могут зависеть от кинематических параметров.

В практическом коде удобнее хранить семейство как базовый интегранд F_0(x,p) и дискретную метку

    label = (n_1,...,n_N,m_1,...,m_M),

так что

    F_label = F_0 * prod_i x_i^n_i * prod_l G_l^m_l.

Задача редукции: для заданного целевого интеграла J(target_label) найти представление

    J(target_label) = sum_a C_a(p) J(label_a),

где C_a(p) — рациональные функции внешних параметров p = (epsilon, s, t, r, ...), а все J(label_a) являются локально конечными интегралами при epsilon = 0. На уровне вывода это должно быть представление исходного интеграла через набор подынтегральных выражений локально конечных интегралов и коэффициентов перед ними.

Важно: равенство обычно является равенством интегралов, а не точечным равенством подынтегральных выражений. Разность исходного интегранда и суммы локально конечных интеграндов является полной дивергенцией плюс, возможно, комбинацией алгебраических тождеств семейства.


2. Два базовых источника соотношений
------------------------------------

2.1. Обычные параметрические IBP

На положительном ортанте можно формально писать

    0 = ∫ d^N x ∂_{x_k} [ Q_k(x) F_label(x) ],

или в векторной форме

    0 = ∫ d^N x div[ Q(x) F_label(x) ].

При раскрытии производной получается линейная комбинация интегралов того же семейства. Если Q_k — полином, то производная по x_i даёт:

    ∂_{x_i} F_label / F_label
      = (nu_i + n_i - 1)/x_i
        + sum_l (mu_l + m_l) (∂_{x_i}G_l)/G_l.

Последний член содержит деление на G_l. После разложения ∂G_l по мономам это приводит к сдвигам m_l -> m_l - 1. Поэтому обычная coordinate-IBP редукция в общем случае живёт на решётке размерности N + M: сдвигаются и степени параметров x_i, и степени полиномов G_l.

Такие соотношения полезны и в универсальном коде их нужно поддерживать, но они требуют аккуратного контроля поверхностных членов.


2.2. Алгебраические тождества по полиномам G_l

Из тождества

    G_l(x) = sum_a c_{l,a}(p) x^a

следует

    J(n,m) = sum_a c_{l,a}(p) J(n+a, m-e_l),

где e_l — единичный сдвиг по l-му полиномиальному индексу. Эти строки не являются IBP, а являются чисто алгебраическими соотношениями между метками семейства. В полной системе они играют роль, близкую к dimension-shift relations в традиционной редукции: позволяют переходить между разными m-слоями.

Практически algebraic rows нужно генерировать всегда: они дешёвые, разреженные и часто критически нужны для закрытия системы.


3. Редукция в фиксированном m-слое через касательные поля
---------------------------------------------------------

Чтобы не сдвигать m-индексы, строятся полиномиальные векторные поля

    Q(x) · ∂ = sum_i Q_i(x) ∂_{x_i},

удовлетворяющие условию

    Q · ∂G_l = H_l(x) G_l,        l = 1,...,M,

где H_l(x) — полиномы.

Тогда

    Q · ∂ log G_l = H_l,

и при раскрытии дивергенции

    0 = ∫ d^N x div[ Q(x) F_label(x) ]

не появляется деления на G_l. Следовательно, m не сдвигается. Раскрытая строка имеет вид

    0 = ∫ F_label [
          div Q
          + sum_i Q_i (nu_i+n_i-1)/x_i
          + sum_l (mu_l+m_l) H_l
        ].

Если разложить Q_i, div Q и H_l по мономам, получается линейная комбинация J(n+shift,m) только внутри фиксированного m-слоя.

Геометрический смысл: такие Q — логарифмические/касательные векторные поля к дивизору G_1 ... G_M = 0. Для одного G_l они находятся как сизигии

    Q_1 ∂_1G_l + ... + Q_N ∂_NG_l - H_l G_l = 0.

То есть нужно найти элементы модуля

    Syz(∂_1G_l, ..., ∂_NG_l, G_l),

а затем отбросить последнюю компоненту. Для нескольких G_l требуется пересечение соответствующих модулей. В коде это можно делать либо через CAS уровня Singular/Sage, либо через линейный ansatz конечной степени над конечным полем.


4. Почему нельзя игнорировать поверхностные члены
-------------------------------------------------

Параметрическая область интегрирования имеет границы:

    x_i = 0,
    x_i = ∞,

а также торические границы, возникающие при совместных масштабированиях нескольких переменных. Поэтому формальное равенство

    ∫ ∂_{x_i}(...) = 0

не всегда верно как равенство интегралов. Оно верно только если primitive/flux не даёт вклада на соответствующих границах.

Это принципиальное отличие между:

    algebraic identity for integrands

и

    valid IBP identity for integrals.

Нужно проверять именно второе. Линейный решатель может быть полностью корректным, но если в систему попали строки с ненулевыми boundary terms, результат редукции будет физически неверным. Поэтому surface-free фильтр является обязательной частью алгоритма, а не постобработкой.

Практическое правило:

    1. Для coordinate IBP вида ∂_{x_i}(Q_i F) проверяются coordinate-boundaries именно этой компоненты:

           x_i -> 0,
           x_i -> ∞.

       Нельзя требовать исчезновения по всем торическим лучам одновременно для каждой coordinate primitive: это слишком строгий критерий и выбрасывает допустимые строки.

    2. Для общего tangent/vector-field IBP вида div(QF) проверяется нормальный поток через торические boundary rays. Для луча rho нужно оценивать valuation выражения

           rho · Q / x

       вместе с valuation интегранда.

    3. Для выбранной аналитической области регулятора, например epsilon -> 0^-, допустимы строки, у которых primitive исчезает в регулируемой области. Но это не означает, что финальный мастер локально конечен при epsilon=0.


5. Локальная конечность
-----------------------

Локальная конечность — это свойство интеграла при epsilon = 0. Её нельзя подменять обычной сходимостью при Re epsilon < 0 или Re epsilon > 0.

Рабочее определение для кода:

    Интеграл локально конечен, если после любого допустимого граничного масштабирования

        x_i ~ lambda^(rho_i),     lambda -> 0,

    pullback интегранда с мерой имеет вид

        lambda^(kappa_rho(epsilon)-1) * bounded_function,

    и при epsilon = 0 выполнено

        kappa_rho(0) > 0

    для всех релевантных boundary rays rho.

Если kappa_rho(0) = 0, а сходимость появляется только из-за знака epsilon-регулятора, такой интеграл не локально конечен. Именно эта ошибка легко приводит к ложным “локально конечным” мастерам, которые на самом деле имеют логарифмическую расходимость при epsilon=0.

Для полиномов с положительными коэффициентами релевантные rho удобно получать из нормальных конусов граней ньютоновых многогранников полиномов G_l и координатных факторов. В более общей ситуации с произвольными знаками коэффициентов такой тест остаётся необходимым, но может не быть достаточным из-за возможных внутренних нулей полиномов; тогда требуется отдельная проверка bulk singularities и assumptions на область параметров.


6. Структура линейной редукции
------------------------------

После генерации строк получается разреженная линейная система

    A(p) · J = 0,

где J — конечный список меток label. Редукция целевого интеграла — это вычисление normal form выбранной переменной относительно строк A.

Ключевые требования к ranking:

    1. Целевой интеграл должен быть исключаемой переменной, если система это позволяет.
    2. Нелокально конечные метки должны исключаться раньше локально конечных.
    3. Среди локально конечных меток свободными должны оставаться наиболее простые интегранды.
    4. Простота не должна определяться только суммой индексов; нужно учитывать реальную сложность множителя:

           степень числителя,
           число полиномиальных знаменателей,
           глубину m-сдвигов,
           размер monomial support,
           наличие внешних параметров в коэффициентах.

Нельзя считать редукцию успешной, если formal normal form найден, но хотя бы один оставшийся term не локально конечен. В таком случае статус должен быть failure/partial, а не success.


7. Конечные поля и восстановление коэффициентов
-----------------------------------------------

Работа напрямую с рациональными функциями от epsilon и кинематических переменных быстро приводит к взрыву выражений. Поэтому основной вычислительный контур должен быть modular/finite-field.

Схема:

    1. Построить шаблоны строк на уровне целочисленных сдвигов и sparse-полиномов.
    2. Выбрать простое p и рациональную точку внешних параметров, не зануляющую знаменатели.
    3. Специализировать коэффициенты в GF(p).
    4. Выполнить sparse row reduction над GF(p).
    5. Получить normal form в этой точке.
    6. Повторить для многих точек параметров и/или многих простых p.
    7. Восстановить коэффициенты C_a(p) как рациональные функции через CRT + rational reconstruction + multivariate interpolation.
    8. Проверить восстановленные C_a(p) на независимых точках.

Для одного epsilon достаточно univariate rational reconstruction. Для нескольких параметров нужен multivariate interpolation: Zippel-style sparse interpolation, Ben-Or/Tiwari для разреженных структур, либо adaptive dense rational ansatz с контролем степеней. В любом случае требуется независимая валидация, иначе возникает риск overfit.


8. Что считать корректным результатом
-------------------------------------

Результат считается корректным только если выполнены все условия:

    1. Все использованные IBP-строки прошли surface-free проверку.
    2. Algebraic rows построены из фактического sparse-разложения полиномов G_l.
    3. Целевой label редуцирован: target является pivot или явно выражен через free variables.
    4. Все free variables в normal form локально конечны при epsilon=0.
    5. Коэффициенты C_a(p) восстановлены как рациональные функции от внешних параметров, а не только найдены в одной числовой точке.
    6. Восстановление проверено на независимых finite-field samples.
    7. Wolfram-like текстовый output синтаксически валиден и позволяет напрямую вставить полученную формулу в дальнейший notebook.

Дополнительно желательно иметь один из следующих сертификатов:

    - divergence certificate:

          F_target - sum_a C_a F_a = div(Q_cert F_base)

      в формате, проверяемом символьной дифференциацией;

    - modular certificate: список primes/samples и hash нормальной формы;

    - numerical check: прямое численное сравнение исходного интеграла и RHS для безопасной евклидовой точки параметров.


9. Типовые ошибки, которые метод должен предотвращать
-----------------------------------------------------

Ошибка 1. Использовать формальные total derivatives без проверки boundary terms.

    Последствие: линейная система становится сильнее, чем должна быть, и может давать красивую, но неверную редукцию.

Ошибка 2. Считать интеграл локально конечным, если он сходится только при epsilon ≠ 0.

    Правильный тест: local_finite(label) должен проверять base_score > 0 при epsilon = 0.

Ошибка 3. Удалять IBP-строки только потому, что в них встречаются промежуточные нелокально конечные интегралы.

    Это неверно. Строка может быть допустимой, а промежуточные интегралы могут быть исключены в ходе редукции. Требование локальной конечности относится к финальной normal form, а не ко всем промежуточным labels.

Ошибка 4. Оставлять простые, но расходящиеся интегралы свободными переменными.

    Ranking должен сначала выдавливать нелокально конечные метки, а уже потом оптимизировать простоту оставшегося локально конечного basis.

Ошибка 5. Восстанавливать коэффициенты по одной finite-field точке.

    Это даёт только числовую редукцию. Для задачи с произвольными epsilon и кинематическими параметрами нужен слой rational-function reconstruction.

Ошибка 6. Смешивать pointwise identity и identity after integration.

    Выходной RHS — это сумма интегралов, а не обязательно сумма интеграндов, равная исходному интегранду точечно.


10. Минимальный пример механизма
--------------------------------

Пусть

    J(n1,n2;m) = ∫ dx dy x^(nu1+n1-1) y^(nu2+n2-1) (1+x+y)^(mu+m).

Для G = 1+x+y поле

    Q = (x y, -x y)

удовлетворяет

    Q · ∂G = xy - xy = 0.

Тогда

    0 = ∫ [ ∂_x(xy F) - ∂_y(xy F) ],

и после раскрытия

    0 = (nu1+n1) J(n1,n2+1;m) - (nu2+n2) J(n1+1,n2;m).

Здесь m не сдвигается, потому что Q касательно к G=0. Это простейшая иллюстрация того, зачем нужны tangent fields.


11. Практическая архитектура метода
-----------------------------------

Полный рабочий pipeline должен состоять из следующих слоёв:

    A. Parser/frontend:
       Wolfram-like/Mathematica-style text input -> canonical symbolic family.

    B. Sparse algebra core:
       sparse polynomials, monomial exponent tuples, derivative tables, valuation tables.

    C. Boundary geometry:
       coordinate boundary tests, toric/valuation local-finiteness tests, Newton polytope rays.

    D. Row generation:
       coordinate IBP rows, algebraic rows, tangent/syzygy IBP rows, optional symmetry rows.

    E. Linear algebra:
       modular sparse Gaussian elimination/RREF, ranking, normal form extraction.

    F. Parameter reconstruction:
       finite-field sampling, CRT, rational reconstruction, multivariate interpolation.

    G. Output/export:
       Wolfram-like текстовые coefficients, integrands, labels, assumptions, diagnostics.

    H. Verification:
       row validity, local finiteness, independent samples, optional numerical or divergence certificate.


12. Relation to the motivating paper
------------------------------------

В arXiv:2605.17057 метод используется как практический способ для проблемных регионов: после упрощений некоторые интегралы не удавалось вычислить напрямую, поэтому выполнялась IBP-редукция в параметрическом представлении и дальнейшая работа велась с locally finite integrals, пригодными для HyperInt.

Для нашего кода это означает, что конечная цель — не просто reduce-to-masters, а reduce-to-locally-finite-integrands. Именно это должно быть главным критерием успешности.
